* SYCL context
#+begin_quote
A context represents the runtime data structures and state required by a SYCL backend API to interact
with a group of devices associated with a platform.
#+end_quote
- [[https://registry.khronos.org/SYCL/specs/sycl-2020/html/sycl-2020.html#sec:context-class][SYCL 2020: 4.6.3. Context class]]

#+begin_quote
Contexts: any SYCL backend resource that is acquired by the user is attached to a context.
A context contains a collection of devices that the host can use and manages memory objects that can be shared between the devices.
Devices belonging to the same context must be able to access each other’s global memory using some implementation-specific mechanism.
A given context can only wrap devices owned by a single platform. A context is exposed to the user with a sycl::context object.
#+end_quote
- [[https://registry.khronos.org/SYCL/specs/sycl-2020/html/sycl-2020.html#_backend_resources_managed_by_the_sycl_application][SYCL 2020: 3.7.1.1. Backend resources managed by the SYCL application]]

#+begin_src cpp
  int main() { ​
      // Pick all GPUs on the first platform ​
      auto platform = sycl::platform::get_platforms().front(); ​
      auto devices  = platform.get_devices(sycl::info::device_type::gpu); ​

      sycl::context ctx{devices};​
      // USM allocation is scoped to the context, not a single device. ​
      // Every device in `ctx` can access this pointer. ​
      int *shared = sycl::malloc_shared<int>(1024, devices[0], ctx);​
      for (auto &d : devices) { ​
          sycl::queue q{ctx, d};​
          q.parallel_for(1024, [=](sycl::id<1> i) { shared[i] += 1; }).wait(); ​
      }​

      sycl::free(shared, ctx);   // freed against the context, not the device ​
  }
#+end_src
** Backend interoperability
SYCL 2020 allows the creation of a sycl::context from a backend native object or to retrieve one from an existing sycl::context.

  cl_context externalClCtx = /* from another component */; ​
  sycl::context ctx = ​

  ​
  // SYCL - native: hand the L0 driver the underlying ze_context ​
  auto ze_ctx = sycl::get_native<sycl::backend::ext_oneapi_level_zero>(ctx); ​
  // call ze_* APIs directly with ze_ctx

* Liboffload improvements
Today liboffload keeps a global =AllocInfoMap= and a sorted =AllocBases= guarded by a
mutex so it can recover (Device, Type, Base, End) from any pointer, plus a 50-try
retry loop in =olMemAlloc= to avoid handing back a pointer that overlaps another
plugin's allocation.

* Future changes
** Splitting olMemAlloc
Currently, =olMemAlloc(Device, Type, Size, **Out)= is one entry point for all
three allocation kinds and takes a =Device= for each. For =OL_ALLOC_TYPE_HOST= the Device parameter is misleading -
every device (from the same plugin) will have access to this allocation,
but the API still requires it and =olGetMemInfo(OL_MEM_INFO_DEVICE)= returns whatever was passed in.

Splitting per kind makes the signature match what each pool can actually support:
#+begin_src cpp
  // ol_result_t olMemAlloc(ol_context_handle_t, ol_device_handle_t, ol_alloc_type_t, size_t, void**);​
  ol_result_t olMemAllocDevice(ol_context_handle_t, ol_device_handle_t, size_t, void**);​
  ol_result_t olMemAllocHost(ol_context_handle_t, /* no Device handle */, size_t, void**);​
  ol_result_t olMemAllocShared(ol_context_handle_t, ol_device_handle_t, size_t, void**);​
#+end_src
=olMemAllocHost= stops accepting a device argument, mirroring SYCL APIs.

** Plugins without native context
With the PR, AMDGPU / CUDA / host plugins do not need any further changes -
the thin =*PluginContextTy= subclasses (forwarding =allocate=/=deallocate=
to =GenericDeviceTy=, =getAllocInfo= via native pointer-query) are enough
for liboffload to satisfy SYCL requirements.

But we can do some extra optimizations:
- Pin host memory (=cuMemHostRegister= / =hsa_amd_memory_lock=) only to the
  devices that are actually in the context, instead of registering globally.
- Reuse the context's native handle (CUDA =CUcontext=) instead of always
  switching to the device's primary context. [[https://github.com/intel/llvm/blob/b0e5431aed9e1e97f28ac7a1a5a7f8f4e5b54ab2/unified-runtime/source/adapters/cuda/context.hpp#L30-L80][Example]]
